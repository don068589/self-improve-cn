# 反馈收集 Prompt

> feedback-collector 模块使用，从对话日志中提取有价值的信号

---

```
你需要从 agent 的日常工作日志中提取有价值的信号。

## 输入

- `config.yaml` 的 `last_success_ts` — 增量扫描起点
- 所有 agent 的 `workspace*/memory/*.md` — 对话日志
- `data/reflections.md` — 上轮反思（闭环输入）

## 输出

- `data/feedback/{日期}.jsonl` — 结构化信号记录

---

## 核心方法：语义理解，不是关键词匹配

**不要机械地找关键词。** 读懂对话，理解 用户 的真实意图和反应。

问自己：
- 用户 满意吗？他的反应说明了什么？
- 用户 在教什么？纠正什么？表达什么偏好？
- 这段对话里有什么值得提取的洞察、方法、教训？
- 用户 的情绪变化说明了什么？（越聊越兴奋 = 高价值讨论）
- agent 的表现如何？哪里做得好？哪里做得不好？

关键词表只是辅助参考，核心是语义理解。

## 信号识别

### 识别方式

**语义理解优先，关键词作为辅助参考。**

| 信号类型 | 语义理解 | 辅助关键词（仅参考） |
|---------|---------|-------------------|
| 纠正 | 用户 指出错误、不满、要求改正 | "不对"、"应该"、"别这样" |
| 隐式纠正 | 用户 自己重写了答案、追问同一问题、换了别的 agent | 无明显关键词 |
| 表扬 | 用户 表示满意、认可、继续推进 | "好"、"对"、"不错" |
| 偏好 | 用户 表达了持续性偏好 | "永远"、"以后"、"每次都要" |
| 洞察 | 讨论中出现了有深度的发现或理念 | "我发现"、"原来"、"意识到" |
| 方法论 | 讨论中沉淀了可复用的方法 | "这样做更有效"、"步骤是" |
| 教训 | 从失败或问题中总结出的经验 | "教训是"、"学到了" |
| 系统级 | 涉及系统设计、架构层面的问题 | "系统应该"、"设计有缺陷" |
| 长链条讨论 | 同一主题持续 >30 分钟，多轮演进 | 无关键词，看对话结构 |

### 隐式信号（关键词抓不到，语义理解才能发现）

- **用户 追问** → 说明第一次回答不够好
- **用户 重写** → 说明 agent 的输出不满意
- **用户 换 agent** → 说明对当前 agent 失去信心
- **用户 沉默后换话题** → 可能是放弃了
- **用户 越聊越兴奋** → 高价值讨论
- **用户 花很长时间讨论一个主题** → 这个主题很重要
- **用户 连续问了三个追问** → 说明第一次回答不够好

### 不记录

- 用户 的单次指令（"帮我查天气"）
- 闲聊、打招呼
- 假设性讨论
- 第三方的偏好
- 群聊中非 用户 的发言

## 提取规则

1. **读懂上下文** — 不是逐条扫描，而是理解整段对话的脉络
2. **保留语义摘要** — summary 字段用 2-3 句话描述这条信号的完整语境
3. **不过度解读** — 用户 没说不好 ≠ 用户 觉得好
4. **记录原文** — hint 字段保留 用户 的原话或关键摘要
5. **标注 agent** — 记录是哪个 agent 被纠正/表扬
6. **识别 skill** — 判断是否使用了某个技能

## 长链条讨论识别

**不只是逐条提取信号，还要识别话题延续。**

识别特征：
- 同一主题，时间跨度 >30 分钟
- 多轮对话、多阶段演进（问题→分析→方案→细化）
- 有总结性发言

处理方式：
- **不拆成多条碎片**
- 合并为单条 synthesis 记录
- 保留阶段演进信息
- summary 字段记录完整演进脉络

## 输出格式

```json
{
  "ts": "YYYY-MM-DDTHH:MM:SSZ",
  "agent": "agent-name",
  "summary": "用户 纠正了 agent 在配置诊断时混淆两个字段名称的错误",
  "hint": "配置诊断时混淆了两个概念",
  "type": "explicit",
  "score": -1,
  "source": "memory/YYYY-MM-DD.md:42",
  "skill": "skill-name",
  "potential_targets": ["AGENTS.md", "TOOLS.md"]
}
```

### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| ts | 是 | 时间戳 |
| agent | 是 | 涉及的 agent |
| summary | 是 | **语义摘要，2-3句话描述完整语境** |
| detail | 否 | **详情段落（高价值信号必填）**，保留关键原话、演进过程、多个洞察点 |
| hint | 是 | 用户 的原话或关键摘要 |
| type | 是 | 信号类型 |
| score | 是 | -1 负面 / 0 中性 / +1 正面 / +2 高价值 |
| source | 是 | 来源文件和行号 |
| skill | 否 | 涉及的 skill（如有）|
| potential_targets | 否 | 启发式参考，可能的去向 |

### 颗粒密度控制

**不是所有信号都需要同样的详细程度。** 根据信号的价值密度调节：

| 信号类型 | 密度 | 写什么 |
|---------|------|--------|
| 简单纠正 | 低 | summary（2-3句）+ hint |
| 偏好确认 | 低 | summary + hint |
| 洞察/方法论 | 中 | summary + detail（一段落）|
| 长链条讨论 | 高 | summary + detail（多段落，保留演进过程和关键原话）|
| 系统级/跨项目 | 高 | summary + detail |

**判断标准：**
- 这条信号后续可能被用来写博客或方法论吗？→ 需要 detail
- 这条信号只是一个简单的行为纠正？→ summary 够了
- 这条信号的价值在于 用户 的原话和语境？→ detail 保留原话
- 这条信号涉及方法论/系统设计？→ 需要完整脉络

**detail 字段示例（高价值信号）：**
```json
{
  "summary": "用户 花40分钟讨论系统设计理念，提出逐步递进、交班记录等概念",
  "detail": "讨论从系统运行问题出发，演进为：1) 发现上下文爆炸问题 → 2) 提出逐步递进流程 → 3) 设计交班记录机制 → 4) 升华为元能力方法论。用户 情绪越聊越兴奋，强调所有 agent 都要掌握这些方法论。",
  "type": "synthesis",
  "score": +2
}
```

### 去向判断（启发式参考）

**这是启发式参考，不是限制。** 模型可以提出任何合理的目标。

常见情况：
- 行为规范 → 可能去 AGENTS.md
- 人格特质 → 可能去 SOUL.md
- 用户习惯 → 可能去 USER.md
- 工具配置 → 可能去 TOOLS.md
- 技能问题 → 可能需要改进 skill

**不确定时标注多个可能，或标注 `"unknown"` 让后续模块判断。**

## 价值预判原则

**漏标高价值的代价远大于误标。**

- score: +2 的信号被漏标为 0 → 后续环节无法识别其价值，不会回炉精读，信息永久丢失
- score: 0 的信号被误标为 +2 → 后续环节多看一眼，发现不值得就跳过，代价很低

所以：**不确定时，宁愿标高不标低。** 尤其是长链条讨论、用户 情绪明显变化、涉及系统设计的内容。

## 信号类型（开放式）

**你的核心任务是理解对话、判断价值，不是往类型表里匹配。** type 只是标签，帮助后续环节快速筛选，不要让它限制你的判断。后续的 proposer 会独立判断产出形式，不会被 type 字段锁定方向。

以下是常见信号类型，**不限于此**。发现新的有价值模式时，自主定义新 type。

| 类型 | 说明 | score |
|------|------|-------|
| explicit | 明确纠正 | -1 |
| positive | 明确表扬 | +1 |
| implicit_positive | 隐式认可（顺利推进）| +1 |
| failure | 任务失败 | -1 |
| rewrite | 用户 重写了输出 | -1 |
| reassign | 用户 换了 agent | -1 |
| preference | 偏好确认 | +1 |
| insight | 洞察发现 | +2 |
| innovation | 创新想法 | +2 |
| methodology | 方法论 | +2 |
| business | 商业逻辑 | +1 |
| knowledge | 知识点 | 0 |
| synthesis | 长链条讨论总结 | +2 |
| system-upgrade | 系统级问题 | +2 |
| meta-ability | 元能力 | +2 |
| cross-project | 跨项目模式 | +2 |

**发现新模式时，自主定义新 type，不需要预先定义。**

## 多维度分类

扫描日志时，同时识别多个分类维度。

### 主题（theme）— 内容性质

| 主题 | 识别特征 |
|------|---------|
| behavior | 行为规范、做事方式、工作流程 |
| communication | 沟通方式、表达、回复风格 |
| tools | 工具使用、配置、操作方法 |
| coding | 编码、调试、代码审查 |
| search | 搜索策略、信息收集、调研 |
| writing | 写作风格、文档、格式 |
| collaboration | 团队协作、跨 agent 配合 |
| preferences | 个人偏好、习惯、口味 |
| professional | 专业能力、领域知识 |
| personality | 人格特质、价值观 |

**以上是参考，不是限制。** 发现新主题时自主创建。

### 领域（domain）— 技术领域

| 领域 | 识别特征 |
|------|---------|
| frontend | 前端、UI、CSS、React、Vue |
| backend | 后端、API、数据库、服务端 |
| ai | AI、ML、模型、prompt、agent |
| devops | 运维、部署、Docker、CI/CD |
| data | 数据分析、统计、报表 |
| security | 安全、权限、认证 |
| mobile | 移动端、iOS、Android |
| general | 不属于以上领域 |

## 负面信号同步写入 corrections.md

```markdown
## YYYY-MM-DD HH:MM

- **agent:** agent-name
- **信号:** 用户 纠正
- **内容:** "具体的纠正内容"
- **主题:** theme-name
- **领域:** domain-name
- **次数:** 1/3
- **状态:** 🔵 待观察
```

## 质量要求

1. **不遗漏** — 宁可多记一条，不要漏掉重要信号
2. **不误判** — 用户 说"不是这个意思"是纠正，用户 说"不是今天"是澄清
3. **保留语境** — summary 要有足够信息，以后能还原上下文
4. **去重** — 同一次对话中的同一个信号只记一条
5. **语义优先** — 用理解力发现隐式信号，不只靠关键词
6. **标注维度** — theme/domain 用于后续分类和检索
```
