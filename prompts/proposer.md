# 产出判断 Prompt

> proposer 模块使用，通盘考虑所有中间产物，判断产出形式，标记高价值

---

```
你需要通盘考虑所有中间产物，判断每条物料的最终去向。

## 核心方法：理解价值，不是匹配通道

**不要机械地想"这条内容符合哪个通道"。** 先理解这条内容的核心价值，再决定怎样才能最大化利用这个价值。

问自己：
- 这条内容的核心价值是什么？
- 怎样才能最大化利用这个价值？
- 只有一种利用方式吗？还是可以多角度利用？
- 这条内容对谁有价值？（仅团队内部 / 有普适价值）

## 输入范围

通盘考虑以下所有中间产物：

| 输入 | 作用 | 优先级 |
|------|------|--------|
| `data/themes/` | 所有分类后的内容（含低频但高质量的洞察） | **最高 — 价值判断的主要来源** |
| `data/hot.md` | ≥3 次的活跃规则 | 高 — 专门服务于规则固化判断 |
| `data/errors/` | 已有的错误知识点，检查重复模式 | 中 — 回炉检查 |
| `data/lessons/` | 已有的经验教训，检查共性 | 中 — 回炉检查 |

**themes/ 是主要输入源。** hot.md 只服务于"规则固化"这一个产出通道。其他产出通道（博客、方法论、技能改进等）的判断主要来自 themes/。

**不要被 hot.md 误导优先级。** hot.md 中的内容是高频规则，适合固化；但低频、高价值的洞察可能只出现 1 次，却比出现 10 次的简单纠正更值得深度处理。

## 价值判断（核心能力）

### 判断原则

1. **理解问题本质** — 先搞清楚问题是什么，再想解决方案
2. **思考最好的处理方式** — 不局限于预设选项
3. **可以提出任何建议** — 包括改配置、改流程、新建文件、删除文件、合并规则等
4. **不确定时标注多个可能** — 审批时和 用户 讨论
5. **不要被 feedback 的 type 字段限制** — type 只是步骤1的初步标签，你需要独立判断产出形式。一条标记为 `type: explicit`（纠正）的信号，背后可能有更深的模式值得写方法论。一条 `type: methodology` 的信号，可能同时有规则固化的价值。type 是参考信息，不是分流指令。

### 价值维度

| 维度 | 低 | 中 | 高 |
|------|----|----|-----|
| 复用性 | 仅当前场景 | 团队内可复用 | 有普适价值 |
| 影响范围 | 单个 agent | 多个 agent | 系统层面 |
| 信息密度 | 单条纠正 | 多条关联 | 长链条讨论凝练 |
| 持久性 | 临时适用 | 长期有效 | 基础原则 |

### value_density 利用

classifier 在 themes/ 中标注了 value_density（low/medium/high）。利用这个信息：
- **high** → 优先处理，考虑多产出通道
- **medium** → 正常判断
- **low** → 规则固化即可（如果达到次数要求）

## 产出形式判断

### 输出通道

| 产出 | 文件 | 条件 |
|------|------|------|
| 规则固化 | proposals/PENDING.md | occurrences ≥ 3 或 用户 明确表达 |
| 技能改进 | proposals/PENDING.md (skill-improvement) | 低分 ≥ 3，skill 可优化 |
| 博客文章 | drafts/blog-{主题}.md | 有普适价值 |
| 方法论 | /path/to/learned/methodologies\ | 可复用思维框架 |
| 错误知识点 | data/errors/ | 有原因和解决方案 |
| 经验教训 | data/lessons/ | "学到了"但不够固化 |
| 系统改进 | proposals/PENDING.md (system-upgrade) | 影响系统架构 |
| 高价值标记 | data/high-value/ | value_density=high |

### 第一步：理解价值

读完一条内容后，先问：
- 这条的核心价值是什么？
- 它能解决什么问题？
- 谁需要它？

### 第二步：选择最佳产出

| 价值类型 | 常见产出 | 判断依据 |
|---------|---------|---------|
| 行为规范 | 规则固化 → PENDING.md | 重复出现 ≥3 次，或 用户 明确表达 |
| 技能问题 | 技能改进 → PENDING.md (skill-improvement) | 低分记录 ≥3，skill 本身可优化 |
| 普适洞察 | 博客文章 → drafts/blog-{主题}.md | 有普适价值，可帮助他人 |
| 可复用方法 | 方法论 → /path/to/learned/methodologies\ | 可复用的思维框架 |
| 具体错误 | 错误知识点 → data/errors/ | 有原因和解决方案 |
| 经验教训 | 教训 → data/lessons/ | "学到了..."但还不够固化 |
| 商业相关 | 商业洞察 → /path/to/learned/business\ | 商业逻辑、盈利模式 |
| 系统设计 | 系统改进 → PENDING.md (system-upgrade) | 影响系统架构 |

### 第三步：一条内容，多角度利用

**不要把一条高价值内容只放到一个通道。** 思考：
- 行为规范 → 规则固化
- 思维方法 → 方法论
- 普适价值 → 博客
- 具体操作 → 知识点

比如一场长链条讨论可以同时产出：
- 规则固化（行为规范部分）
- 博客文章（设计理念部分）
- 方法论（思维框架部分）

**具体思考示例：**

假设有一条信号：长链条讨论，涉及系统设计方法论和具体的行为规范。

思考过程：
1. 这条的核心价值是什么？→ 系统设计方法论 + 具体的行为规范
2. 谁需要它？→ 团队内部（行为规范）+ 外部（方法论有普适价值）
3. 只有一种利用方式吗？→ 不是。至少三种：
   - **规则固化**：某项原则 → 写入 AGENTS.md 作为工作流规范
   - **方法论**：设计框架 → 写入 methodologies/ 作为可复用框架
   - **博客**：主题文章 → 写入 drafts/ 作为对外分享
4. 拆还是合？→ 拆。三种产出各自独立审批，更灵活

## 规则固化判断

### 常规固化（需要重复验证）

1. **出现次数 ≥ 3** — 同一纠正/规则被记录 3 次以上
2. **用户 明确表达** — 用户 说过"永远"、"绝不"、"以后"等确定性词语
3. **影响范围广** — 适用于所有 agent，不是个别场景

### 紧急固化（跳过次数要求）

当 corrections.md 中标记了 `severity: high`，或满足以下条件时，跳过"3 次重复"要求，标注 `[URGENT]`：

1. 造成了实际损害
2. 跨 agent 协调失败
3. 用户 明确要求固化

紧急固化只是跳过等待重复的门槛，**审批流程不变**——仍然写入 PENDING.md，等 用户 裁决。

### 不固化的情况

- 只出现 1-2 次的纠正（继续观察）
- 特定项目/场景的临时规则
- 用户 的假设性讨论
- 第三方偏好

## 目标文件判断（核心能力）

**proposer 的核心任务是判断"这条物料应该去哪里"。**

### 常见情况参考（启发式，不是限制）

| 内容性质 | 常见目标 | 举例 |
|---------|---------|------|
| 行为规范、工作流程 | AGENTS.md | "应该先X再Y"、"不要X" |
| 人格特质、态度风格 | SOUL.md | "要更直接"、"人格应该" |
| 用户习惯、偏好观念 | USER.md | "用户 喜欢X"、"用户 不喜欢Y" |
| 工具使用、环境配置 | TOOLS.md | "新装了X"、"用X工具" |
| 个人偏好、私人规则 | MEMORY.md | "用户 的个人习惯" |
| 角色定位、身份描述 | IDENTITY.md | "这个agent的角色是" |
| 环境信息、通讯录 | 共享目录 registry | "新机器人"、"新服务端口" |
| 技能使用问题 | skill改进 | "skill X 有问题" |
| 新能力需求 | 新skill建议 | "需要X能力" |
| 系统设计问题 | SYSTEM.md/ENGINE.md | "系统应该" |
| 配置问题 | openclaw.json | "contextWindow 太小" |

**这只是参考，不是限制。**

### 判断方法

问自己：
1. **问题本质是什么？** — 先理解清楚
2. **最好的解决方案是什么？** — 不局限于上面的表
3. **需要改什么？** — 可能是文件、配置、流程、结构
4. **有更好的建议吗？** — 可以提出任何想法

### 示例

**问题：模型上下文超限**

✗ 机械判断："加规则到 AGENTS.md"
✓ 正确判断："这是配置问题，建议改 openclaw.json 的 contextWindow 或换模型"

**问题：agent 之间任务冲突**

✓ 可以判断："加协作规则到 AGENTS.md"
✓ 也可以判断："创建任务认领机制，需要新建文件"
✓ 也可以判断："这是系统问题，建议加任务调度功能"

**多种方案都可以，在 PENDING.md 里写清楚，审批时讨论。**

## errors/ 和 lessons/ 回炉

**errors/ 和 lessons/ 不是终点，是中间站。**

每次运行时，检查已有的 errors/ 和 lessons/：

| 发现 | 处理 |
|------|------|
| 多条 error 有共性 | 提炼为规则 → PENDING.md |
| lesson 重复出现 | 升级为规则固化 → PENDING.md |
| error/lesson 可泛化 | 提炼为方法论 → methodologies/ |
| error/lesson 有普适价值 | 提炼为博客 → drafts/ |

**闭环：errors/lessons 既是产出位置，也是下一轮的输入源。**

## 高价值标记

通盘考虑 themes/（主要）和 hot.md 后，标记高价值项：

| 标准 | 说明 |
|------|------|
| 影响范围广 | 适用于多个 agent 或系统层面 |
| 可提炼性强 | 能提炼为方法论、博客文章 |
| 价值密度高 | 信息量大，经过多轮对话凝练 |
| value_density = high | classifier 已标注为高密度 |

标记为 `high_value` 的项写入 `data/high-value/`，供步骤6回炉精读。

## 元能力：提出更好的处理建议

**proposer 不仅是判断去向，还要能提出更好的处理建议。**

在生成建议时，主动思考：

1. **这个问题是否有更好的解决方案？**
   - 不局限于"加规则"，可能改配置、改流程、新建文件更好
   - 例：上下文超限 → 不是加规则让 agent 小心，而是改 contextWindow 配置

2. **是否应该创建新文件而不是追加到现有文件？**
   - 如果现有文件已经太长，拆分可能更好
   - 如果是全新的能力需求，新建 skill 可能更合适

3. **是否应该拆分成多条建议分别处理？**
   - 一个长链条讨论可能涉及多个层面
   - 拆分后每条独立审批，更灵活

4. **是否应该先观察而不是立即固化？**
   - 有些模式需要更多数据验证
   - 标注"建议观察"而不是"建议固化"

5. **是否有其他系统/工具能更好地处理这个问题？**
   - 系统设计层面的问题不应该用规则补丁
   - 可能需要 system-upgrade 而不是 rule-addition

6. **这条内容能不能帮助其他人？**
   - 有普适价值 → 考虑博客或方法论
   - 仅内部适用 → 规则固化就够

如果有更好的建议，在 PENDING.md 中增加字段：

```markdown
- **备选方案:** {其他可能的处理方式}
- **建议:** {你认为最好的处理方式及理由}
```

## 输出格式

### 规则固化 → PENDING.md

```markdown
## [P-YYYYMMDD-NNN] 简短标题

- **来源:** corrections.md #行号 / hot.md:行号 / themes/{主题}
- **建议修改:**
  - 当前状态：（目标文件中现有的相关内容，如果有）
  - 建议添加/修改：（具体的新内容或修改方案）
- **目标文件:** {具体路径，如 openclaw.json 或 AGENTS.md}
- **目标说明:** 为什么建议去这里
- **理由:** 出现 N 次 / 用户 明确表达 / 紧急原因
- **事件涉及:** agent: X, Y, Z（谁出了这个问题）
- **建议适用:** 全团队 / 特定 agent: X, Y（这条规则该给谁）
- **备选方案:** （如果有更好的处理建议，或多个可能的目标）
- **状态:** 🟡 待审批
```

**目标文件写完整路径或具体文件名**，方便审批时定位和讨论。
**备选方案可选**，如果有更好的处理建议或多个可能的目标就写上。
**目标可以是任何合理的建议**，包括：
- 系统文件（AGENTS.md, SOUL.md, USER.md, TOOLS.md, MEMORY.md, IDENTITY.md）
- 配置文件（openclaw.json, config.yaml）
- 共享目录（{shared_root}/registry/）
- skill 文件（~/.openclaw/skills/xxx/SKILL.md）
- 新建文件
- 删除/合并文件
- 其他任何合理的处理方式

### 紧急固化 → PENDING.md

```markdown
## [P-YYYYMMDD-NNN] [URGENT] 简短标题

- **来源:** corrections.md（severity: high）
- **紧急原因:** 造成了实际损害 / 跨 agent 协调失败 / 用户 要求
...其余字段与常规格式相同...
```

### 技能改进 → PENDING.md

```markdown
## [P-YYYYMMDD-NNN] 改进 {skill名} 技能

- **类型:** skill-improvement
- **目标文件:** {workspace_root}\skills\{skill名}\SKILL.md
- **来源:** themes/{主题}/{规则}.md（frontmatter.skill 字段，低分 ≥ 3）
- **发现的问题:** {问题模式}
- **建议修改:**
  在 SKILL.md 中追加：
  > {具体改进内容}
- **涉及记录:**
  - {日期} {agent} {hint摘要}
- **来源追溯:**
  - 主题：{theme}
  - 领域：{domain}（可选）
  - 项目：{project}（可选）
- **状态:** 🟡 待审批
```

**目标文件写完整路径，方便审批后定位执行。**

**技能改进的判断标准：**
- themes/ 中 frontmatter.skill 字段存在
- 低分记录 ≥ 3
- hint 中有可改进的模式（不是 agent 个人问题，是 skill 本身可以优化）

### 系统改进 → PENDING.md

```markdown
## [P-YYYYMMDD-NNN] 系统改进：{改进点}

- **类型:** system-upgrade
- **来源:** 跨项目讨论 / 长链条讨论
- **发现的问题:** {结构性问题描述}
- **改进建议:** {具体改进方案}
- **影响范围:** {哪些模块/文件需要改}
- **理由:** 为什么这是系统级问题
- **状态:** 🟡 待审批
```

**与普通固化的区别：**

| 普通固化 | 系统级改进 |
|---------|-----------|
| 影响 agent 行为 | 影响系统设计 |
| 写入 AGENTS.md/TOOLS.md | 写入 openclaw.json、MODULE.md、prompt 文件 |
| 规则层面的改变 | 结构层面的改变 |

**系统级改进的目标文件：**

| 改进类型 | 目标文件 |
|---------|---------|
| 新增模块 | modules/*/MODULE.md, config.yaml |
| 修改流程 | prompts/*.md |
| 配置变更 | config.yaml, openclaw.json |
| 系统架构 | SYSTEM.md, ENGINE.md |

### 博客草稿 → drafts/

```markdown
# {标题}

> 草稿状态：待润色
> 来源：self-improve 运行 {run_id}
> 发现时间：{时间}
> 核心价值：{一句话说明为什么值得写}

## 核心观点

{提炼的核心观点，3-5 条}

## 正文

{文章主体，可以是：}
{- 问题背景}
{- 解决思路}
{- 实践案例}
{- 总结升华}

## 待润色项

- [ ] 标题是否吸引人
- [ ] 开头是否抓住读者
- [ ] 结尾是否有升华
- [ ] 案例是否充分
- [ ] 语言是否流畅
```

**应该写博客的情况：**
- 有普适价值，不只适用于本团队
- 值得对外传播的想法、经验
- 可以帮助其他人的方法论
- 技术实践、踩坑总结

**不应该写博客的情况：**
- 纯内部规则
- 涉及隐私或敏感信息
- 太碎片化，没有完整观点

## 写入质量要求

1. **用目标文件的语言风格** — 读取目标文件，模仿其格式和语气
2. **具体可执行** — 不写模糊的建议，写具体的规则
3. **标注来源** — 每条建议都标注出处和出现次数
4. **不重复** — 检查 PENDING.md 中是否已有类似建议
